# Frontend architecture: Leptos (WASM) + Deno static host

## Current direction

| Layer | Stack |
|--------|--------|
| **UI** | **Leptos** CSR in `ui/` — Rust → **WASM** via **Trunk** + `wasm-bindgen`. Semantic DOM from Leptos; no React/Vue/Svelte app. |
| **Static host** | **Deno** `scripts/serve_ui_dist.ts` serves `ui/dist` with correct `application/wasm` MIME. This is the only Deno surface for the web UI: **no** client app logic in JavaScript/TypeScript. |
| **Legacy** | `viewer-egui/` (egui canvas) remains until features are ported into Leptos. |

## “Server-side WASM” vs browser WASM

- **Browser:** Leptos output (`*.wasm` + glue `*.js` generated by wasm-bindgen) runs in the tab. That is the primary WASM execution for rendering.
- **Deno (optional later):** Shared Rust logic can be compiled to a **WebAssembly module** and instantiated in Deno with `WebAssembly.instantiate` for server-side verification (e.g. schedule bytes) **without** shipping a second UI stack. That is a separate build target from the Leptos bundle; document when you add it.

## Security notes (same as before)

- WASM does not remove XSS if you inject untrusted HTML alongside the app; keep CSP and trusted HTML only.
- Do not embed signing keys or issuance secrets in the client bundle.
- CORS and HTTPS for API calls to the token node / authority remain mandatory.

## SSR

Leptos **SSR** (server-rendered HTML + hydrate) usually uses **cargo-leptos** / Axum, not Deno. If you need SSR later, either introduce a Rust HTTP server for HTML shell or evaluate `leptos` SSR integration explicitly—do not assume Deno can render Leptos server-side without a dedicated integration.
